## Обзор

Скилл для Claude Code, который полностью автоматизирует развёртывание VPN-сервера на базе 3x-ui. Вы даёте свежий VPS (IP-адрес и root-пароль от провайдера) — скилл делает всё остальное: защищает сервер, устанавливает панель управления, настраивает прокси и помогает подключиться через клиент Hiddify.

Создан специально для новичков. Не нужно разбираться в Linux, SSH или сетевых протоколах — достаточно запустить Claude Code и описать задачу.

## Возможности

- 🔒 **Полная защита сервера** — SSH-ключи, файрвол UFW, fail2ban, hardening ядра
- 📦 **Установка панели 3x-ui** — со случайно сгенерированными учётными данными
- ⚡ **VLESS TCP + Reality** — домен не нужен, рекомендуется для большинства
- 🌐 **VLESS TCP + TLS** — с доменом, панель доступна напрямую без SSH-туннеля
- 🕸️ **VLESS XHTTP + Reality** — SplitHTTP транспорт, максимальная защита от DPI
- 🎭 **Сайт-заглушка NebulaDrive** — реалистичная страница облачного хранилища на Nginx
- 🔑 **SSH на кастомном порту** — порт 22 закрывается, SSH переносится на выбранный порт
- ♻️ **Автообновление сертификата** — cron-скрипт открывает порт 80, обновляет, закрывает — ежедневно в 03:00
- ☁️ **WARP outbound** — маршрутизирует Google/YouTube через Cloudflare, убирает гео-детект и капчи
- 📱 **Инструкции по Hiddify** — пошаговое руководство для клиента
- 🖥️ **Удалённый и локальный режим** — через SSH или прямо на сервере
- ✅ **Пошаговые проверки** — критические точки верифицируются на каждом этапе
- 👻 **ICMP отключён** — сервер не отвечает на ping
- 🚪 **Умное управление firewall** — открыты только необходимые порты (443 для VPN, кастомный SSH, панель если домен)

## Варианты протоколов

| | Path A | Path B | Path C |
|--|--------|--------|--------|
| **Транспорт** | TCP | TCP | XHTTP (SplitHTTP) |
| **Безопасность** | Reality | TLS | Reality |
| **Домен для VPN** | Нет | Да (обязателен) | Нет |
| **Сложность** | Простой | Средний | Простой |
| **Сайт-заглушка** | Нет | Да (Nginx) | Нет |
| **Защита от DPI** | Высокая | Высокая | Очень высокая |
| **Доступ к панели** | SSH-туннель (или домен, если настроен) | Напрямую через домен | SSH-туннель (или домен, если настроен) |
| **Рекомендуется** | Новичкам | При наличии домена | Максимальная скрытность |

> **Path A** рекомендуется большинству пользователей. **Path C** (XHTTP) труднее всего обнаружить и заблокировать, но требует актуальной версии клиента.
>
> **Домен необязателен для Path A и C** — если домен есть, доступ к панели через домен настраивается независимо от выбранного протокола VPN.

## Что нового в этом форке

### Панель через домен (опционально для Path A/C, обязательно для Path B)
- **Path B (TLS):** Домен ТРЕБУЕТСЯ для VPN протокола. Панель доступна через домен: `https://ваш-домен:{panel_port}/{web_base_path}`
- **Path A (Reality) / Path C (XHTTP):** Домен ОПЦИОНАЛЬНО. При наличии домена панель доступна напрямую вместо SSH туннеля.

SSL-сертификат, UFW и автообновление настраиваются в отдельном шаге ПОСЛЕ выбора протокола.

### Автообновление сертификата (только если домен настроен)
Порт 80 по умолчанию закрыт. Скрипт `/root/cert-renew.sh` управляет файрволом автоматически: открывает порт 80 → обновляет сертификат → закрывает порт 80 → перезапускает x-ui. Запускается ежедневно в **03:00** через cron. Лог: `/var/log/cert-renew.log`.

### SSH на кастомном порту (опционально для всех путей)
Опционально: переместить SSH с порта 22 на выбранный порт для дополнительной безопасности. UFW сначала открывает новый порт — только потом закрывает 22, исключая риск блокировки. Работает независимо от выбора протокола VPN или домена.

### Сайт-заглушка NebulaDrive
Nginx отдаёт реалистичную страницу облачного хранилища с тёмной темой. Обычный посетитель видит настоящий сайт, а не ошибку подключения. Шрифты JetBrains Mono + Russo One, анимированные градиенты.

### WARP outbound (обход гео-детекта Google)
После того как VPN работает, скилл предлагает опциональный шаг: маршрутизировать Google, YouTube и сервисы Google AI (Gemini, AI Studio) через Cloudflare WARP. Вместо IP датацентра Google видит IP Cloudflare — исчезают капчи и гео-блокировки. Остальной трафик идёт напрямую, скорость не падает.

### XHTTP + Reality (Path C)
Новый вариант протокола: SplitHTTP транспорт с безопасностью Reality. Разбивает трафик VPN на множество мелких HTTP-запросов — одна из самых сложных для обнаружения конфигураций.
- Домен не нужен
- Без `xtls-rprx-vision` flow (не поддерживается с XHTTP)
- Sniffing с `routeOnly: true` — браузеры работают без сбоев
- Требует Xray core v1.8.16+

## Workflow

```
Свежий VPS (IP + root + пароль)
  |
  +-- Часть 1: Закаление сервера
  |   +-- Генерация SSH ключей
  |   +-- Обновление системы
  |   +-- Непривилегированный пользователь + sudo
  |   +-- Файрвол UFW (SSH + 443 только)
  |   +-- Закаление ядра (sysctl)
  |   +-- BBR TCP оптимизация
  |   +-- ICMP отключён (скрытность)
  |   +-- Ярлык конфига SSH
  |
  +-- Часть 2: Установка VPN (выбери путь)
  |   +-- Установка 3x-ui панели
  |   |
  |   +-- [Path A] TCP + Reality
  |   |     +-- Reality SNI сканер
  |   |     +-- Создание inbound через API
  |   |     +-- Панель через SSH туннель
  |   |
  |   +-- [Path B] TCP + TLS (домен)
  |   |     +-- SSL сертификат (acme.sh)
  |   |     +-- Панель через домен (прямо)
  |   |     +-- Cron продление сертификата (03:00)
  |   |     +-- SSH переведён на кастомный порт
  |   |     +-- NebulaDrive stub site
  |   |
  |   +-- [Path C] XHTTP + Reality
  |         +-- Reality SNI сканер
  |         +-- Создание XHTTP inbound через API
  |         +-- Панель через SSH туннель
  |
  +-- Ссылка подключения + инструкция Hiddify
  +-- Шаг 21a (Опционально): WARP Outbound
  |     +-- Установка warp-cli
  |     +-- Настройка режима прокси
  |     +-- Добавление Xray outbound + routing rule
  |     +-- Тест через SOCKS5
  |
  +-- fail2ban + блокировка SSH
  +-- Готово: Защищённый сервер + работающий VPN
```

## Firewall и управление портами

Скилл реализует **минимальное открытие портов** — открыты только необходимые:

| Порт | Статус | Назначение | Когда используется |
|------|--------|-----------|-------------------|
| 22 | Варьируется | SSH | Закрыт по умолчанию при использовании кастомного порта (Шаг 22); перемещён на {ssh_port} |
| 443 | ✅ Всегда открыт | VLESS VPN (все пути) | Требуется для всех протоколов (A, B, C) |
| 80 | ❌ Закрыт (авто-открыт) | ACME обновление сертификата | Открывается автоматически через крон в 03:00 на 5 минут, потом закрывается (только Шаг 14c) |
| {panel_port} | Варьируется | Панель управления 3x-ui | Закрыт по умолчанию; открывается только если настроен домен (Шаг 14c); защищен SSL + пароль |
| 40000 | Только локальный | WARP SOCKS5 прокси | **НЕ открыт в интернет** — работает только на 127.0.0.1 (опционально, Шаг 21a) |
| {ssh_port} | ✅ Открыт (кастомный) | SSH доступ | Создаётся если переместить SSH с порта 22 (Шаг 22) |

### Ключевые моменты безопасности:

- **Порт 80** обычно ЗАКРЫТ. Открывается автоматически только во время ежедневного обновления сертификата (03:00) через крон `/root/cert-renew.sh`, затем закрывается.
- **Порт 40000** (WARP) **никогда НЕ открыт в интернет** — работает только на localhost (127.0.0.1). Если видишь 0.0.0.0:40000, что-то неправильно.
- **Порт панели** защищен SSL-сертификатом (если домен) и требует пароль для входа. Включи двухфакторную аутентификацию для дополнительной защиты.
- **fail2ban** защищает SSH от перебора после завершения настройки (устанавливается на Шаге 22).

### Проверка портов после настройки:

```bash
# Проверить правила UFW
ssh {nickname} "sudo ufw status numbered"

# Проверить какие порты слушают
ssh {nickname} "sudo ss -tlnp | grep -E ':443|:80|:40000|:{panel_port}'"

# Убедиться что порт 80 НЕ открыт постоянно
ssh {nickname} "sudo ufw status | grep 80"  # Должно быть пусто или "deny"

# Убедиться что WARP только на localhost (если установлен)
ssh {nickname} "sudo ss -tlnp | grep 40000"  # Должно быть 127.0.0.1:40000
```

## Решение проблем

### Проблемы с SSH и подключением

| Проблема | Решение |
|----------|---------|
| `Permission denied (publickey)` | Проверь права ключа: `chmod 700 ~/.ssh && chmod 600 ~/.ssh/*` |
| `Host key verification failed` | Удали старый ключ: `ssh-keygen -R <ip-сервера>` |
| SSH timeout или отказ подключения | Если используешь кастомный порт (Шаг 22): `ssh -p {custom_port} {username}@{server-ip}` |

### Проблемы с панелью

| Проблема | Решение |
|----------|---------|
| Панель недоступна (без домена) | Используй SSH-туннель: `ssh -L {panel_port}:127.0.0.1:{panel_port} {nickname}` → открой `https://127.0.0.1:{panel_port}/{web_base_path}` |
| Панель недоступна (с доменом) | Проверь: 1) домен указывает на сервер (`nslookup {domain}`), 2) UFW разрешает порт ({panel_port}): `sudo ufw status`, 3) сертификат существует: `sudo ls /root/cert/{domain}/`, 4) Xray запущен: `sudo x-ui status` |
| Забыл пароль панели | Сбрось пароль: `ssh {nickname} "sudo x-ui setting -reset"` |

### Проблемы с VPN подключением

| Проблема | Решение |
|----------|---------|
| Reality не подключается | Пересканируй SNI через сканер; убедись что твой домен НЕ используется как SNI |
| XHTTP ошибка клиента | Обнови Hiddify/Nekobox/v2rayN до последней версии (XHTTP требует свежих клиентов) |
| Интернет не работает после подключения VPN | Проверь логи панели: `ssh {nickname} "sudo x-ui log"` — ищи TLS ошибки или проблемы с привязкой |

### Проблемы с сертификатом

| Проблема | Решение |
|----------|---------|
| Сертификат не обновился | Проверь крон: `ssh {nickname} "sudo crontab -l \| grep cert"` — должно быть `0 3 * * * /root/cert-renew.sh`. Запусти вручную: `ssh {nickname} "sudo /root/cert-renew.sh"` и проверь `/var/log/cert-renew.log` |
| Порт 80 не закрывается после обновления | Проверь вывод скрипта: `ssh {nickname} "tail /var/log/cert-renew.log"`. Если застрял, закрой вручную: `ssh {nickname} "sudo ufw deny 80/tcp"` |

### Проблемы с firewall и портами

| Проблема | Решение |
|----------|---------|
| **Порт 80 постоянно открыт (КРИТИЧНО)** | Так НЕ должно быть. Порт 80 должен быть закрыт кроме как во время обновления сертификата. Проверь: `sudo ufw status \| grep 80` (должно быть пусто или "deny"). Если открыт, закрой: `sudo ufw deny 80/tcp` |
| **Порт 40000 открыт в интернет (КРИТИЧНО)** | WARP SOCKS5 должен слушать ТОЛЬКО на localhost. Проверь: `sudo ss -tlnp \| grep 40000`. Если показывает `0.0.0.0:40000` или не-127.0.0.1, WARP неправильно настроен. Переустанови: `sudo apt remove cloudflare-warp && sudo apt install cloudflare-warp` |
| UFW показывает неожиданно открытые порты | Запусти: `sudo ufw status numbered` и сравни с ожидаемым списком (SSH, 443, и {panel_port} если домен) |

### Проблемы с WARP

| Проблема | Решение |
|----------|---------|
| `warp-cli` не найдена | Установи: `ssh {nickname} "sudo apt update && sudo apt install -y cloudflare-warp"` |
| Порт 40000 не слушает | Проверь режим: `ssh {nickname} "warp-cli mode"` — должно быть `Proxy`. Переконфигурируй: `ssh {nickname} "warp-cli --accept-tos mode proxy && warp-cli --accept-tos connect"` |
| `warp=off` в trace | WARP подключена но не маршрутизирует. Переключи режим: `warp-cli --accept-tos mode proxy` и переподключись |
| Google всё ещё показывает IP датацентра | Проверь routing rule в панели: inbound=`inbound-443`, outbound=`warp-cli`, domains содержат `geosite:google`, `geosite:youtube`. Перезапусти: `sudo x-ui restart` |

| Проблема | Решение |
|----------|---------|

| Ошибка клиента с XHTTP | Обновите Hiddify/Nekobox/v2rayN до последней версии |
